変更差分に含まれる共通化できそうなコードをチェックします。

## 使い方

```
/check:commonize
```

## 手順

### 1. 変更ファイルの取得

```bash
git diff --name-only HEAD
```

変更ファイルが存在しない場合は「変更差分がありません」と出力して終了。

### 2. 重複・類似コードの検出

変更差分および変更ファイル全体を解析し、以下のパターンを検出する。

**検出対象:**
- 同じロジックが複数箇所に書かれている（コピペコード）
- 類似した関数・メソッドが複数ファイルに存在する
- 同じ条件分岐が繰り返されている
- 同じ定数・マジックナンバーが複数箇所に散在している

### 3. 共通化候補の分類

**🔴 高優先度（強く共通化を推奨）**
- 完全に同一のコードブロックが3箇所以上
- ビジネスロジックの重複

**🟡 中優先度（共通化を検討）**
- 類似したコードブロックが2箇所以上
- 同じ定数・設定値が複数ファイルに散在

**🔵 低優先度（参考情報）**
- パターンは似ているが文脈が異なる
- 将来的に分岐する可能性がある

### 4. 共通化方法の提案

検出した候補に対して、適切な共通化方法を提示する。

**提案パターン:**
- ユーティリティ関数への抽出 (`utils/`)
- カスタムフック化 (`hooks/useXxx.ts`)
- 共通コンポーネント化
- 定数ファイルへの集約 (`constants/`)
- 型定義の共通化 (`types/`)

## 除外設定

以下はチェック対象外:
- `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`
- `node_modules/**`
- `dist/**`, `build/**`, `.next/**`
- バイナリファイル（画像、フォントなど）
- テストファイル内の意図的な重複（`*.test.ts`, `*.spec.ts`）

## 注意事項

- 意図的な重複（文脈が異なる・将来的に分岐する可能性がある）は低優先度として報告
- 自動リファクタリングは提案のみ、実行は確認後に行う
- 共通化によって逆に複雑になる場合は提案しない
