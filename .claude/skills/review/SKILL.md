ブランチの変更内容をサブエージェントで並列チェックし、人間のレビューを支援するレポートを生成する。

## 入力パターン

- `/review` - 現在のブランチのmainとの差分をレビュー
- `/review TASK-XXX` - 指定タスクに関連する変更をレビュー（タスクのスコープ・完了条件も参照）

## 前提確認

- mainブランチとの差分またはステージング済み変更が存在すること（どちらもなければ「レビュー対象がありません」と伝えて終了）

## 手順

### 1. 変更内容の収集（内部処理）

以下を実行し、変更の全体像を把握する。

```bash
git diff --cached --stat          # ステージング済みの変更ファイル一覧
git diff --cached                 # ステージング済みの差分詳細
git diff main...HEAD --stat       # コミット済みの変更ファイル一覧
git diff main...HEAD              # コミット済みの差分詳細
git log main..HEAD --oneline      # コミット一覧
```

上記を統合して「レビュー対象の全差分」「変更ファイルリスト」を構成する。

タスクIDが指定されている場合は `docs/tasks/TASK-XXX.md` を読み、タスクのスコープ・完了条件をレビュー観点に含める。

**出力**: 変更ファイルリスト、差分詳細、コミット一覧 → 手順2〜4の入力

### 2. レビュールールの収集（内部処理）

以下のファイルを探索し、存在するものをすべてレビュー基準として読み込む。

| ファイル | 内容 |
| --- | --- |
| `.coderabbit.yaml` | CodeRabbitのレビュールール |
| `CLAUDE.md` | プロジェクトのコードスタイル・規約 |
| `eslint.config.*` / `.eslintrc.*` | ESLint設定（ルートのみ） |
| `tsconfig.json` | TypeScript設定 |
| `docs/designs/` | 該当する設計書 |
| `docs/specs/` | 該当する仕様書 |

**出力**: レビュー基準の集合 → 手順3・4のチェック観点

### 3. カスタムサブエージェントによる並列レビュー

以下の3つのカスタムサブエージェントを `Agent` ツールで**並列に**起動する。各エージェントには変更ファイルリストと差分詳細を渡す。

**1. code-reviewer**（コード品質チェック）

渡すプロンプト:
```
以下の変更ファイルをレビューしてください。
変更ファイル: [変更ファイル一覧]
差分詳細: [差分の内容]

チェック観点: 型安全性、エラーハンドリング、命名規則、コード規約違反、セキュリティ脆弱性、パフォーマンス問題
報告形式: 各指摘を Error / Warning / Info に分類し、ファイル:行番号と説明を返す
```

**2. doc-checker**（設計書・仕様書との整合性チェック）

渡すプロンプト:
```
以下の変更ファイルと設計書・仕様書を比較してください。
変更ファイル: [変更ファイル一覧]
設計書: [該当する設計書パス]（存在しない場合は「設計書なし」と報告）
仕様書: [該当する仕様書パス]（存在しない場合は「仕様書なし」と報告）

報告形式: 整合 / 差異あり（差異の内容）で返す
```

設計書・仕様書の特定方法: `docs/designs/` と `docs/specs/` 配下のファイル名・見出しを確認し、変更ファイルの機能領域と一致するものを渡す。該当なしの場合は「なし」と指定する。

**3. impact-analyzer**（変更影響の分析）

渡すプロンプト:
```
以下の変更が他のファイルや機能に与える影響を分析してください。
変更ファイル: [変更ファイル一覧]

分析観点: 変更ファイルを import しているファイル、型の変更による影響、APIの互換性
報告形式: 影響を受けるファイル・機能のリストを返す。影響なしの場合は「影響なし」
```

### 4. 逐次チェック

手順3と**並列に**以下の5つのチェックを順番に実行する。

**判定基準（全チェック共通）:**

| レベル | 基準 |
|--------|------|
| **Error** | バグの原因になる問題、セキュリティ脆弱性、ビルド・型エラー |
| **Warning** | 規約違反、保守性の低下、パフォーマンス懸念 |
| **Info** | スタイル改善の提案、参考情報 |

1. **タイポチェック** — `/check-typo` スキルの SKILL.md を読み、その手順に従い実行
2. **表記ゆれチェック** — 差分の追加行から以下のパターンを検出:
   - 日本語: ユーザー vs ユーザ / サーバー vs サーバ / データベース vs DB
   - 英語: color vs colour / center vs centre
   - 記号: 〜 vs ～ / ー vs −
   - 単位: 10px vs 10 px
   - プロジェクト全体で Grep し、多数派への統一を推奨する（Info として報告）
3. **影響範囲チェック** — `/check-diff-impact` スキルの SKILL.md を読み、その手順に従い実行
4. **共通化チェック** — 変更差分と変更ファイル全体を解析:
   - 高（Warning）: 完全同一コードが3箇所以上 / ビジネスロジックの重複
   - 中（Info）: 類似コードが2箇所以上 / 設定値の散在
   - 低（Info）: パターンは似ているが文脈が異なる
5. **不要なexportチェック** — 差分の `export` 宣言を抽出し、プロジェクト内の `import` を Grep で検索:
   - 参照なし → Warning として報告
   - 除外: `export default`、`index.ts` の再エクスポート、`package.json` の `exports` 対象

### 5. レビューレポートの統合

手順3・4の全結果を統合し、以下のフォーマットで報告する。各セクションが0件の場合は「なし」と記載する。

```
コードレビューレポート

対象: main...<ブランチ名> (<コミット数>コミット, <変更ファイル数>ファイル)
タスク: TASK-XXX（該当する場合。なければ省略）

─────────────────────────
Error（修正必須）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
Warning（修正推奨）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
Info（参考情報）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
ドキュメント整合性
─────────────────────────
- 設計書: [整合 / 差異あり（内容） / なし]
- 仕様書: [整合 / 差異あり（内容） / なし]

─────────────────────────
変更影響
─────────────────────────
- [影響を受けるファイル・機能]（or 影響なし）

─────────────────────────
タイポ・表記ゆれ
─────────────────────────
- タイポ: [件数]件
- 表記ゆれ: [件数]件

─────────────────────────
影響URL
─────────────────────────
- [影響を受けるURL]（or なし）

─────────────────────────
共通化候補
─────────────────────────
- 高: [件数]件 / 中: [件数]件 / 低: [件数]件（or なし）

─────────────────────────
不要なexport
─────────────────────────
- [ファイル:行番号] `export [名前]` — 参照箇所なし（or なし）

─────────────────────────
総合判定: [LGTM / 要修正 / 要確認]
─────────────────────────
```

**総合判定の基準:**
- **LGTM**: Error が 0件
- **要修正**: Error が 1件以上
- **要確認**: Error は 0件だが Warning が 5件以上、またはドキュメント差異あり

### 6. 問題がある場合のアクション提案

Errorがある場合:
```
修正するには:
  /code-fix
```

- `/review` - 現在のブランチの変更をレビュー
- `/review TASK-XXX` - 指定タスクのスコープに沿ってレビュー
