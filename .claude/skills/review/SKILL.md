ブランチの変更内容をサブエージェントで並列チェックし、人間のレビューを支援するレポートを生成する。

## 入力パターン

- `/review` - 現在のブランチのmainとの差分をレビュー
- `/review TASK-XXX` - 指定タスクに関連する変更をレビュー（タスクのスコープ・完了条件も参照）

## 前提確認

- mainブランチとの差分またはステージング済み変更が存在すること（どちらもなければ「レビュー対象がありません」と伝えて終了）

## 手順

### 1. 変更内容の収集（内部処理）

以下を実行し、変更の全体像を把握する。

- `git diff --cached --stat` でステージング済みの変更ファイル一覧を取得
- `git diff --cached` でステージング済みの差分の詳細を取得
- `git diff main...HEAD --stat` でコミット済みの変更ファイル一覧を取得
- `git diff main...HEAD` でコミット済みの差分の詳細を取得
- 上記を統合してレビュー対象の全差分とする
- `git log main..HEAD --oneline` でコミット一覧を取得
- タスクIDが指定されている場合、`docs/tasks/TASK-XXX.md` を読む

### 2. レビュールールの収集（内部処理）

以下のファイルを探索し、存在するものをすべてレビュー基準として読み込む。

| ファイル                          | 内容                               |
| --------------------------------- | ---------------------------------- |
| `.coderabbit.yaml`                | CodeRabbitのレビュールール         |
| `CLAUDE.md`                       | プロジェクトのコードスタイル・規約 |
| `eslint.config.*` / `.eslintrc.*` | ESLint設定（ルームルートのみ）     |
| `tsconfig.json`                   | TypeScript設定                     |
| `docs/designs/`                   | 該当する設計書                     |
| `docs/specs/`                     | 該当する仕様書                     |

### 3. カスタムサブエージェントによる並列レビュー

以下の3つのカスタムサブエージェント（`.claude/agents/` に定義済み）を `Agent` ツールで**並列に**実行する。

**1. code-reviewer**（コード品質チェック）

```
「以下の変更ファイルをレビューしてください。
 変更ファイル: [変更ファイル一覧]」
```

→ エージェント側にチェック観点・規約参照ルールが定義済み

**2. doc-checker**（設計書・仕様書との整合性チェック）

```
「以下の変更ファイルと設計書・仕様書を比較してください。
 変更ファイル: [変更ファイル一覧]
 設計書: [該当する設計書パス]
 仕様書: [該当する仕様書パス]」
```

→ エージェント側にチェック観点・報告フォーマットが定義済み

**3. impact-analyzer**（変更影響の分析）

```
「以下の変更が他のファイルや機能に与える影響を分析してください。
 変更ファイル: [変更ファイル一覧]」
```

→ エージェント側に分析観点・報告フォーマットが定義済み

### 4. 逐次チェック

手順3と**並列に**以下の4つのチェックを順番に実行する。致命的なエラー（Gitコマンド失敗など）が発生した場合はそこで停止する。

1. **タイポチェック** — `/check:typo` スキルの手順に従い実行
2. **表記ゆれチェック** — 以下の手順で実行
3. **影響範囲チェック** — `/check:diff-impact` スキルの手順に従い実行
4. **共通化チェック** — 以下の手順で実行

#### 表記ゆれチェック

差分の追加行（`+` で始まる行）から以下のパターンを検出する。

**日本語:** ユーザー vs ユーザ / サーバー vs サーバ / データベース vs DB / アプリケーション vs アプリ
**英語:** color vs colour / center vs centre / initialize vs initialise
**記号:** 〜 vs ～ (波ダッシュ vs 全角チルダ) / ー vs − (長音符 vs マイナス記号)
**単位:** 10px vs 10 px / 100% vs 100 %

プロジェクト全体で同じ単語の異なる表記を Grep で検索し、多数派または正式名称への統一を推奨する。`.vscode/settings.json` の `cSpell.words` も参照する。

#### 共通化チェック

変更差分および変更ファイル全体を解析し、以下のパターンを検出する。

**検出対象:**
- 同じロジックが複数箇所に書かれている（コピペコード）
- 類似した関数・メソッドが複数ファイルに存在する
- 同じ条件分岐の繰り返し
- 同じ定数・マジックナンバーが複数箇所に散在

**優先度分類:**
- **高**: 完全同一コードが3箇所以上 / ビジネスロジックの重複
- **中**: 類似コードが2箇所以上 / 設定値の散在
- **低**: パターンは似ているが文脈が異なる / 将来的に分岐する可能性

**共通化提案:** ユーティリティ関数(`utils/`) / カスタムフック(`hooks/useXxx.ts`) / 共通コンポーネント / 定数ファイル(`constants/`) / 型定義(`types/`)

### 5. レビューレポートの統合

手順3・4の全結果を統合し、以下のフォーマットで報告する。

```
コードレビューレポート

対象: main...<ブランチ名> (<コミット数>コミット, <変更ファイル数>ファイル)（ステージング済みを含む）
タスク: TASK-XXX（該当する場合）

─────────────────────────
Error（修正必須）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
Warning（修正推奨）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
Info（参考情報）
─────────────────────────
- [ファイル:行番号] [説明]

─────────────────────────
ドキュメント整合性
─────────────────────────
- 設計書: [整合 or 差異あり（内容）]
- 仕様書: [整合 or 差異あり（内容）]

─────────────────────────
変更影響
─────────────────────────
- [影響を受けるファイル・機能]（or 影響なし）

─────────────────────────
タイポ・表記ゆれ
─────────────────────────
- タイポ: [件数]件
- 表記ゆれ: [件数]件
- 不要な文字変更: [件数]箇所

─────────────────────────
影響URL
─────────────────────────
- [影響を受けるURL]（or なし）

─────────────────────────
共通化候補
─────────────────────────
- 高優先度: [件数]件 / 中: [件数]件 / 低: [件数]件

─────────────────────────
総合判定: [LGTM / 要修正 / 要確認]
─────────────────────────
```

### 6. 問題がある場合のアクション提案

Errorがある場合:

```
修正するには:
  /fix [問題の説明]
```

ドキュメント差異がある場合:

```
ドキュメントを更新する必要があります。
修正後に /commit してください。
```

## $ARGUMENTS

- `/review` - 現在のブランチの変更をレビュー
- `/review TASK-XXX` - 指定タスクのスコープに沿ってレビュー
