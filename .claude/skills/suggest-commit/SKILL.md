現在の変更差分を分析し、コミットメッセージ規約に基づいたコミット名を提案します。

## 使い方

```
/suggest-commit
```

## 手順

### 1. 変更内容の収集

以下のコマンドを順に実行する:

```bash
git diff --cached --stat        # ステージング済みの変更ファイル一覧
git diff --cached               # ステージング済みの差分詳細
```

**ステージング済みの差分がある場合はそちらのみを対象とし、未ステージの差分は無視する。**

ステージング済みの差分がない場合のみ、以下を実行:

```bash
git diff --stat                 # 未ステージの変更ファイル一覧
git diff                        # 未ステージの差分詳細
```

どちらもない場合は「変更がありません」と出力して終了。

スタイル参考として直近のコミットも取得:

```bash
git log --oneline -5            # 直近のコミットメッセージ（スタイル参考）
```

### 2. コミットメッセージ規約の参照

CLAUDE.md のコミットメッセージ規約に従い、プレフィックスを判定する:

| プレフィックス | 用途 |
|--------------|------|
| `feat:` | 新機能 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみ |
| `refactor:` | リファクタリング（機能変更なし） |
| `test:` | テストの追加・修正 |
| `chore:` | ビルド設定・依存パッケージ更新 |

### 3. コミット名の提案

変更内容を分析し、以下のルールで候補を3つ提案する:

**ルール:**
- プレフィックス + 半角スペース + 変更内容の要約
- 要約は日本語で簡潔に（プレフィックスを含めて50文字以内を目安）
- 「何を」「なぜ」が伝わる内容にする

**複数プレフィックスが該当する場合の優先順位:**
1. `fix:` — バグ修正が含まれていれば最優先
2. `feat:` — 新機能が含まれていれば次に優先
3. その他 — 変更の主目的に合わせる

**3候補の出し方:**
- 候補1: 最適と判断したプレフィックス + 最も簡潔な要約
- 候補2: 同じプレフィックスで別の切り口の要約（例: 「何を」重視 vs 「なぜ」重視）
- 候補3: 別のプレフィックスが妥当な場合はそちらで提案。妥当でなければ候補1と同じプレフィックスでより詳細な要約

### 4. 出力

```
コミット名の提案
═══════════════════════════
対象: <ステージング済み or 未ステージ>（<ファイル数>ファイル）

  1. <コミット名候補1>
  2. <コミット名候補2>
  3. <コミット名候補3>

使用するには:
  git commit -m "<選んだコミット名>"
```
