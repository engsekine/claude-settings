# コード修正スキル

コードの修正を指示された際に、変更内容を事前に表示してユーザーの確認を得てから適用する。

## 入力パターン

- `〜を修正して` - 指定された問題を修正
- `〜をfixして` - 指定された問題を修正
- `/fix 〜` - 指定された問題を修正
- レビュー後の修正指示

## 手順

### 1. 問題箇所の特定

指示された内容から、修正が必要なファイルと箇所を特定する。

- ファイルパスが明示されている場合: そのファイルを読む
- ファイルパスが不明な場合: Grep/Glob で該当箇所を検索
- レビュー結果からの修正: レビューレポートに記載されたファイル:行番号を参照

### 2. 修正内容の立案

修正前のコードを読み、以下を確認:

- 修正の影響範囲（同じパターンが他にもあるか）
- コーディング規約への準拠（`rules/coding/` を参照）
- 既存の設計・実装パターンとの整合性

### 3. 変更内容の提示

**必ず以下のフォーマットで変更内容を提示し、ユーザーの確認を求める:**

```
📝 修正内容の確認
═══════════════════════════════

## 修正対象

ファイル: <ファイルパス>
行番号: <行番号>
理由: <修正理由>

## 変更前

\`\`\`<言語>
<修正前のコード>
\`\`\`

## 変更後

\`\`\`<言語>
<修正後のコード>
\`\`\`

## 影響範囲

- <影響を受けるファイル・機能>
- （影響がない場合: この変更による影響はありません）

─────────────────────────
この内容で修正しますか？ (yes/no)
─────────────────────────
```

### 4. ユーザー確認の待機

ユーザーからの応答を待つ:

- `yes` / `y` / `ok` / `修正して` → 修正を実行
- `no` / `n` / `待って` → 修正をキャンセル
- 修正案の提示 → 修正内容を再検討

### 5. 修正の実行

ユーザーが承認した場合のみ、Edit ツールで修正を適用する。

### 6. 修正完了の報告

```
✅ 修正完了

ファイル: <ファイルパス>
変更: <変更内容の要約>

次のステップ:
- テストを実行: npm test
- 動作確認: npm run dev
- コミット: /commit
```

### 7. 修正履歴の記録（オプション）

重要な修正パターンや学びがあった場合、`rules/coding/skills.md` に記録を提案:

```
この修正パターンを skills.md に記録しますか？
- パターン: <修正パターン>
- 理由: <なぜこの修正が必要だったか>
```

## 例外パターン

以下の場合は確認をスキップして即座に修正可能:

- タイポの修正（明らかなスペルミス）
- フォーマッターの適用（Prettier, ESLint --fix）
- ユーザーが明示的に「確認不要」と指示した場合

ただし、これらの場合でも修正後に変更内容を報告すること。

## 注意事項

- 複数ファイルの修正が必要な場合、1ファイルずつ確認を求める
- 影響範囲が大きい場合（3ファイル以上）、事前に全体の修正計画を提示する
- コーディング規約に準拠した修正を行う
- 既存のコードスタイルを尊重する
