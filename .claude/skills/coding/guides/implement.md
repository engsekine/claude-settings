# 新規実装スキル

新しい機能やコンポーネントを実装する際に、既存コードの再利用とコードの適切な配置を確認する。

## 入力パターン

- `〜を実装して` - 新しい機能を実装
- `〜を追加して` - 新しいコンポーネント/機能を追加
- `〜を作成して` - 新しいファイル/モジュールを作成
- `/implement 〜` - 新しい機能の実装

## 手順

### 1. 既存コードの調査

新規実装を始める前に、必ず既存コードベースを調査する。

#### 1.1 類似機能の検索

- **Grep/Glob で類似実装を検索**
    - 同じドメインの機能
    - 似たようなUI/ロジック
    - 共通のパターンやユーティリティ

#### 1.2 再利用可能なコードの特定

以下を確認:

- 既存のユーティリティ関数・Hook
- 共通コンポーネント
- 共有される型定義
- ビジネスロジック

### 2. コード配置の検討

#### 2.1 機能固有コードの確認

既存コードに「機能固有の場所に配置された汎用的なコード」がないか確認:

```
[NG] 悪い例:
  /features/UserProfile/utils/formatDate.ts
  /pages/Dashboard/helpers/calculateTotal.ts

[OK] 良い例:
  /lib/utils/formatDate.ts
  /lib/helpers/calculateTotal.ts
```

#### 2.2 リファクタリングの判断

以下の条件を満たす場合、共通化を提案:

- **2回以上使用される** (DRY原則)
- **ドメイン固有でない** (汎用性がある)
- **独立している** (副作用が少ない)

### 3. 実装計画の提示

**必ず以下のフォーマットで実装計画を提示し、ユーザーの確認を求める:**

```
📋 実装計画
═══════════════════════════════

## 実装内容

機能: <実装する機能>
ファイル: <作成/修正するファイル>

## 既存コードの再利用

### 再利用可能なコード

- [ ] <既存の関数/コンポーネント名> (`<ファイルパス>`)
  理由: <なぜ再利用できるか>

- [ ] <既存の型定義> (`<ファイルパス>`)
  理由: <なぜ再利用できるか>

### リファクタリングが必要なコード

- [ ] `<ファイルパス>:<関数名>`
  問題: 機能固有の場所にあるが汎用的
  提案: `<新しい配置先>` に移動

## 実装の流れ

1. <ステップ1>
2. <ステップ2>
3. <ステップ3>

## 影響範囲

- <影響を受けるファイル・機能>
- （影響がない場合: この実装による既存機能への影響はありません）

─────────────────────────
この計画で実装を進めますか？ (yes/no)
─────────────────────────
```

### 4. ユーザー確認の待機

ユーザーからの応答を待つ:

- `yes` / `y` / `ok` / `実装して` → 実装を開始
- `no` / `n` / `待って` → 実装をキャンセル
- 修正案の提示 → 実装計画を再検討

### 5. 実装の実行

#### 5.1 リファクタリング優先

共通化が必要な場合は、先にリファクタリングを実行:

1. 既存の汎用コードを共通の場所に移動
2. 既存の使用箇所を更新
3. テストを実行して既存機能が壊れていないことを確認

#### 5.2 新規実装

リファクタリング後、新しい機能を実装:

1. 共通化されたコードを活用
2. コーディング規約に準拠
3. 適切な場所に配置

### 6. 実装完了の報告

```
✅ 実装完了

## 実装内容

ファイル: <ファイルパス>
機能: <実装した機能>

## 再利用したコード

- <既存コンポーネント/関数名> (`<ファイルパス>`)
- <既存の型定義> (`<ファイルパス>`)

## リファクタリング

- `<元のパス>` → `<新しいパス>` に移動
  理由: <移動理由>

## 次のステップ

- テストを実行: npm test
- 動作確認: npm run dev
- コミット: /commit
```

## コード配置のガイドライン

プロジェクトタイプ別のガイドライン:

- **フロントエンド**: `.claude/guides/frontend.md` を参照
- **バックエンド**: プロジェクト固有の構造に従う
- **フルスタック**: 各レイヤーのガイドラインを参照

### 汎用的な配置原則

- **再利用可能なコード**: プロジェクトルート近くの共通ディレクトリ
- **機能固有のコード**: 機能/ドメインごとのディレクトリ
- **設定ファイル**: プロジェクトルートまたは専用の `/config` ディレクトリ

## 注意事項

- **過度な抽象化を避ける**: 1回しか使わないコードを無理に共通化しない
- **段階的リファクタリング**: 大規模な移動は避け、小さく段階的に実施
- **既存パターンの尊重**: プロジェクト固有の構造や命名規則を優先

## 例外パターン

以下の場合は既存コード調査をスキップ可能:

- プロトタイプ・PoC の実装
- ユーザーが明示的に「新規作成のみ」と指示した場合
- 完全に新しいドメインの初回実装

ただし、これらの場合でも将来的な共通化の可能性は意識すること。
